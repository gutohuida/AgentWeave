AgentWeave — Canonical Regeneration Specification

A single, self-contained specification detailed enough for any capable LLM to regenerate the entire AgentWeave framework — CLI, Hub backend, Hub UI, and documentation — without access to the original source code.
CLI v0.40.0 Hub v0.33.0 Status: Living Standard Spec-Driven Development Last generated: 2026-07-20

0.Front Matter#

This document is the canonical, authoritative specification for AgentWeave, a multi-agent AI collaboration framework. It is written for spec-only development: the specification — not the code — is the source of truth. A capable large language model (LLM) reading this document must be able to regenerate a functionally-equivalent implementation of the entire project from this document alone.

Note This specification separates normative requirements (binding on any implementation) from informative reference material (the current reference implementation's technology choices). Where the two conflict, normative requirements win. The reference implementation is provided so regeneration can be faithful (byte-comparable behavior) rather than merely equivalent.
Requirement FR-META-001 This document describes a specific implementation revision — CLI v0.40.0 and Hub v0.33.0 (mirrored in the header badges). A regenerated implementation must target these versions unless a later spec revision supersedes them. The specification follows a living-spec / spec-as-source persistence model: this document is the human-authored source of truth and implementation artifacts are treated as derived. Consequently, when a code change alters any normative value set defined here (constants, command surface, REST endpoints, SSE event names, database columns/indexes), the change must be reflected back into this document in the same change set, and the version badges plus the specification changelog updated. Where this document and the code disagree, the discrepancy must be recorded as an Open Issue rather than silently resolved.

0.1How to read this spec#

Read §1 Overview and §2 Glossary first — every later section depends on the terms defined there. Then read the §3 Domain Model, which defines the shared entities (Task, Message, Agent, etc.) that the CLI, MCP tools, Hub backend, and Hub UI all operate on. The four subsystem sections (§4 CLI, §5 Transport, §6 MCP Tools, §7 Hub Backend, §8 Hub UI) can be read independently but all reference §3. Finish with the §12 Regeneration Guide, which gives the recommended build order.

0.2Conformance classes#

Following the WHATWG HTML Living Standard's producer/consumer distinction, this spec separates two kinds of requirement:

  • Requirements on producers (what a user, agent, or client is allowed to supply — e.g. valid field values, allowed inputs). These state what input is well-formed.
  • Requirements on consumers (how AgentWeave software must act in response — validation, persistence, side effects, responses). Requirements on producers have no bearing on how a consumer must handle malformed input; consumer behavior for invalid input is specified separately.

The key words must, must not, should, should not, and may are to be interpreted as in RFC 2119. A requirement written as "must" is binding; "should" is a strong recommendation; "may" is optional.

0.3Typographic conventions#

The following labeled blocks distinguish what kind of text you are reading. Only Requirement blocks and inline must/should/may statements are normative.

Requirement FR-XXX-000 A normative, testable assertion. Every requirement carries a stable ID of the form FR-<AREA>-<NNN> and is listed in the Requirements Index.
Note Non-normative clarification or rationale. Explains the "why" behind a rule; not itself binding.
Example Illustrative only. Concrete values shown are one valid possibility, not the only allowed form.
Warning A hazard, security concern, or common mistake to avoid.
Reference Implementation Informative. Describes the current technology choices (Python, FastAPI, React, etc.). An implementation may substitute an equivalent stack unless a normative requirement says otherwise.
Open Issue A known discrepancy between this document and the reference implementation, or an intentionally-unresolved ambiguity, that a reader must not treat as settled. Each Open Issue states the observed behavior and what remains to be decided or fixed.

Defined terms appear in bold accent color at their definition and are cross-referenced elsewhere. Numbered algorithms specify ordered behavior that must be performed in sequence:

  1. This is an algorithm step.
  2. If a condition holds, do something.
  3. Otherwise, return to step 1.

1.Overview & Scope#

AgentWeave is a framework that lets multiple independent AI coding agents (for example Claude, Kimi, Gemini, Codex, Copilot, and OpenAI-compatible models run through a proxy) collaborate on the same software project. It provides three things: (a) a shared, structured Task and Message substrate so agents can delegate work and communicate; (b) a pluggable transport layer so that substrate can live on the local filesystem, in a git branch, or in a central Hub server; and (c) a web Hub for humans to observe and steer the collaboration in real time.

Requirement FR-CORE-001 The system must consist of two independently-packaged components: a CLI package (installable and usable with zero required runtime third-party dependencies) and a Hub (a server plus web dashboard). The CLI must be fully functional for local, single-machine collaboration without the Hub.
Requirement FR-CORE-002 The CLI must expose both an agentweave and a short-alias aw console entry point that behave identically.

1.1Operation modes#

ModeMechanismBest for
Hub (recommended)Self-hosted server + web dashboard; agents/CLI talk to it over HTTP; real-time updates via SSE.Teams, multi-machine, live monitoring.
Zero-relay MCPMCP tools plus a watchdog daemon that auto-delivers messages/tasks and can auto-trigger agents.Autonomous loops on one machine.
Manual relayThe CLI generates a "relay prompt" the human pastes into the next agent.Quick one-off delegation, no daemon.
Requirement FR-CORE-003 A project's collaboration mode must be one of hierarchical, peer, or review. In hierarchical mode a single principal agent delegates and reviews; in peer mode any agent may assign work to any other; review mode is specialized for code-review workflows.

1.2Non-goals#

The following are explicitly out of scope. A regenerating implementation must not add them unless a future spec revision requires it:

  • AgentWeave does not implement or bundle the agent models/CLIs themselves. It orchestrates existing external agent CLIs (e.g. claude, kimi, codex).
  • It does not provide a hosted/SaaS multi-tenant service. The Hub is single-tenant, self-hosted, and project-scoped by API key.
  • It does not perform automated code review quality judgements itself; it provides the workflow (task states, review gates) that agents and humans use.
  • It does not manage secrets storage. Only environment-variable names are stored in config; secret values live in the shell environment or a git-ignored .env.
  • The CLI does not require the Hub, and the Hub does not require any specific agent CLI to be installed to start.

2.Glossary#

These terms are used with these exact meanings throughout the spec.

Agent — A named AI coding assistant participating in a project. Identified by a name matching ^[a-zA-Z0-9_-]{1,32}$. Any matching name is accepted; a curated list of known agents exists only for defaults/suggestions.

Runner — How an agent's CLI is invoked (e.g. claude, kimi, codex, claude_proxy, manual). Determines the command, session-resume flag, output format, and MCP-registration command.

Principal — The single agent designated to lead a Session in hierarchical mode. At most one agent per project.

Session — The runtime state of a project's collaboration: its mode, principal, agent roster, and task lists. Persisted as JSON.

Task — A unit of delegated work with a title, description, assignee, priority, and a lifecycle status.

Message — An inter-agent communication with a sender, recipient, subject, and content.

Question — A query an agent posts to the human (Hub only), optionally blocking, that the human answers.

Job — A scheduled, recurring instruction to an agent, defined by a cron expression.

Role — A behavioral guide assigned to an agent (e.g. backend_dev, coordinator). An agent may hold multiple roles simultaneously.

Transport — The pluggable backend through which messages and tasks are stored and retrieved: local (filesystem), git (orphan branch), or http (Hub).

Hub — The self-hosted FastAPI server plus React dashboard that centralizes collaboration state and provides real-time monitoring.

Watchdog — A background daemon that polls for new messages/tasks and can auto-trigger agent CLIs.

MCP — The Model Context Protocol. AgentWeave exposes tools to agents via MCP servers (a CLI-side server and a Hub-side server).

Relay prompt — A generated block of text summarizing pending work, meant to be pasted into an agent to hand off control (manual-relay mode).

Proxy agent — An OpenAI/Anthropic-compatible model (e.g. MiniMax, GLM) run through the Claude CLI by overriding its base URL and API key (claude_proxy runner).

Pilot mode — A per-agent flag that disables auto-triggering so a human controls that agent's sessions manually.

Yolo mode — A per-agent flag allowing the agent to act without confirmation prompts.

3.Domain Model & State#

This section defines the entities shared by all subsystems. Field names are normative wire/serialization names. The CLI persists these as JSON files under .agentweave/; the Hub persists them in a relational database (§7.3). Both representations must preserve the fields defined here.

Note Identifiers are generated as <prefix>-<8 hex chars>, e.g. task-1a2b3c4d, msg-9f8e7d6c, session-…, job-…. Timestamps are ISO-8601 strings (UTC). Every ID that becomes a filename must match ^[a-zA-Z0-9_-]+$ (path-traversal defense).

Valid session roles are principal, delegate, reviewer, collaborator.

3.2Agent#

An Agent is a named participant. Per-agent configuration keys are: role, since, runner, env_vars, model, yolo, pilot, runner_options.

Requirement FR-AGT-001 Agent names must be validated against ^[a-zA-Z0-9_-]{1,32}$. A cluster-prefixed form <cluster>.<agent> (used by the git transport) must validate the segment after the final dot.
Reference Implementation Known agent names seeded for defaults/suggestions (not a validation gate): claude, kimi, gemini, codex, aider, cline, cursor, windsurf, copilot, opendevin, gpt, qwen, minimax, glm, opencode. Default roster when none specified: [claude, kimi]. The root context file an agent auto-reads is CLAUDE.md for claude, GEMINI.md for gemini, and AGENTS.md for everything else.

3.4Task lifecycle#

The eight task statuses are pending, assigned, in_progress, completed, under_review, revision_needed, approved, rejected. The canonical happy path and branches:

pending → assigned → in_progress → completed → under_review → approved
                                             ↘ revision_needed (loops back to in_progress)
                                             ↘ rejected
StatusMeaning
pendingCreated, not yet assigned.
assignedAssigned to an agent, awaiting start.
in_progressAgent actively working.
completedWork finished, awaiting review.
under_reviewBeing reviewed.
revision_neededChanges required; loops back to in_progress.
approvedReview passed; done.
rejectedWill not be completed.
Note The status set is enforced everywhere (CLI, MCP, Hub). The transition graph above is the intended workflow; the reference implementation validates the value of status but does not hard-block arbitrary transitions, so tooling should present the graph while the store accepts any valid status value.

3.6Question#

A Question is Hub-only. Fields: id, from_agent, question, answer?, answered (bool), blocking (bool), created_at, answered_at?.

Requirement FR-QST-001 When a question is answered, the implementation must set answered=true, record answered_at, and deliver the answer back to the asking agent as a Message.

3.7Job & JobRun#

A Job schedules a recurring instruction to an agent. Fields: id, name, agent, message, cron, session_mode (new|resume, default new), enabled (default true), created_at, last_run?, next_run?, run_count, last_session_id?, source (hub|local). A JobRun records one firing: id, job_id, fired_at, status (fired|failed), trigger (scheduled|manual), session_id?, error_summary?.

Requirement FR-JOB-001 cron must be a valid 5-field expression (minute hour day month weekday, weekday 0=Sunday). Firing a job must deliver a Message to the target agent, update last_run/next_run, increment run_count, and record a JobRun.

3.8Event#

An Event is a structured log record: event_type, optional agent, arbitrary data, severity (debug|info|warning|error; default info), and timestamp. The CLI appends events to a JSON-lines file; the Hub stores them in a table and rebroadcasts them over SSE (§9, §7.5).

4.CLI Specification#

The CLI (agentweave / aw) is the primary operator interface. It reads agentweave.yml for declarative configuration and persists runtime state under .agentweave/.

Requirement FR-CLI-001 All task and message state mutations must be performed while holding a file-based mutex (a named lock) and must write files atomically (write to a temp file, then rename) so concurrent CLI/watchdog processes cannot corrupt state.
Requirement FR-CLI-002 The CLI must automatically load a .env file from the current working directory. Shell-exported variables must take precedence over .env values.

5.Transport Layer#

All message and task I/O flows through a pluggable transport so the same CLI and MCP behavior works on the local filesystem, across machines via git, or against the Hub.

Requirement FR-TR-001 The transport interface must define these six abstract operations, and every backend must implement all six: send_message(message_data) → bool, get_pending_messages(agent) → list, archive_message(message_id) → bool, send_task(task_data, error?) → bool, get_active_tasks(agent?) → list, and get_transport_type() → "local"|"git"|"http".
Requirement FR-TR-002 The interface must additionally provide optional operations that default to not supported (raising) or no-op on backends that lack them: job operations (create_job, list_jobs, get_job, update_job, delete_job, fire_job), push_session, push_roles_config, push_log, register_session, is_agent_registered, and get_agent_registration.

6.MCP Tools#

Agents interact with AgentWeave through MCP tools. Two MCP servers exist: a CLI-side server (stdio; operates through the active transport) and a Hub-side server (stdio; calls the Hub REST API directly using HUB_URL, HUB_API_KEY, HUB_PROJECT_ID). Tool names and semantics are shared; some tools require Hub/HTTP transport.

Requirement FR-MCP-001 Both MCP servers must expose the messaging and task tools with identical names and argument shapes: send_message(from, to, subject, content, type?, task_id?), get_inbox(agent) (which must auto-archive returned messages), mark_read(message_id), list_tasks(agent?), get_task(task_id), create_task(title, description?, assignee?, assigner?, priority?, requirements?, acceptance_criteria?), and update_task(task_id, status).
Requirement FR-MCP-002 Session tools must include get_status(), list_agents() (returning per-agent name, session_role, runner, dev_roles, is_principal), get_agent_context(agent), and get_context(role). Job tools must include create_job, list_jobs, get_job, toggle_job, delete_job, run_job.
Requirement FR-MCP-003 The following tools must require Hub/HTTP transport and must return an error when only local transport is active: ask_user(from_agent, question, blocking?), get_answer(question_id), register_agent(name, contact_mode, ...) (where contact_mode ∈ {poll, mcp-push, watchdog-spawn}), update_agent_config(name, ...), register_session(from_agent, session_id), get_agent_config(agent), and heartbeat(agent).
Requirement FR-MCP-004 The CLI-side server must additionally provide save_checkpoint(agent, session_intent, files_modified, decisions, next_steps, reason, blockers?, verification_commands?), writing a markdown checkpoint to .agentweave/shared/checkpoints/<agent>-<timestamp>.md.
Note The Hub-side server implements the same tool set by translating each call into the corresponding REST endpoint (§7.4), auto-injecting project_id into payloads. get_inbox is implemented as list-then-PATCH-read per message.

7.2Authentication#

Requirement FR-HUB-003 Except for GET /health and GET /api/v1/setup/token, every endpoint must require a bearer token via Authorization: Bearer <key>. A valid key must start with the exact prefix aw_live_, exist in the key store, and not be revoked. Failures must return HTTP 401. The key resolves the request's project scope.
Requirement FR-HUB-004 The SSE stream must accept either the bearer header or a short-lived signed ticket passed as ?token=. A ticket has the form aw_ticket_{project_id}:{expires}:{signature} where the signature is a truncated (32-char) HMAC-SHA256 over {project_id}:{expires}; expired or invalid tickets must yield 401. Non-SSE endpoints must not accept ?token=.
Requirement FR-HUB-005 GET /api/v1/setup/token must be unauthenticated but reachable only from local/Docker-internal callers — accepted source addresses are IPv4 loopback (127.0.0.0/8), IPv6 loopback (::1), and the Docker private ranges 10.0.0.0/8 and 172.16.0.0/12 (Docker port-forwarding makes host requests appear to originate from the Docker gateway IP). It returns a non-revoked API key as {"api_key": "<the aw_live_… key>"}, 403 for any other caller, or 503 if no key exists. This bootstraps agentweave activate without manual key entry.
Open Issue The reference implementation selects the key with LIMIT 1 and no explicit ordering, so "the" key is only well-defined when a single non-revoked key exists; multi-key deployments get a database-order-dependent result. A future revision should specify a deterministic selection (e.g. oldest non-revoked key).

7.5SSE events#

Requirement FR-HUB-008 The SSE manager must maintain per-project subscriber queues (bounded; dropping events for slow consumers) and broadcast named events. Open Issue compact_request is part of the reserved event vocabulary but is not currently broadcast over SSE by the reference implementation: POST /agents/{name}/compact only persists a compact_request event log and enqueues the agent message. Either the endpoint should broadcast it (to match new_session_request) or the name should be documented as event-log-only. Left unresolved pending a decision.

7.6Scheduler#

Requirement FR-HUB-009 The scheduler must run as an async singleton with a persistent job store, load all enabled jobs on startup, and use UTC with misfire grace and coalescing. On firing a job it must: create a Message to the target agent (sender user, subject "Scheduled job: {name}", content including the job message plus a [Session: …] or [NewSession] marker); update last_run/next_run/run_count; create a JobRun; prune history to the last 100 runs per job; and broadcast job_fired + message_created. On failure it must mark the run failed, store a redacted error_summary (≤500 chars), and broadcast job_run_failed at severity error.
Reference Implementation Backend: Python 3.10+, FastAPI, SQLAlchemy (async), SQLite via aiosqlite by default (PostgreSQL via asyncpg supported), Alembic migrations, APScheduler AsyncIOScheduler + SQLAlchemyJobStore, sse-starlette, Pydantic schemas with extra="forbid". The Hub-side MCP server is run as python -m hub.mcp_server.

8.Hub UI#

The Hub dashboard is a single-page application over the Hub REST API and SSE stream. It lets a human observe agents, tasks, messages, questions, logs, and jobs, and drive agent chats.

8.1 Shell & navigation

Requirement FR-UI-001 On startup the app must bootstrap configuration: load a stored API key or fetch one from GET /api/v1/setup/token; show a "Connecting…" state until ready; then render the shell and open the SSE connection. If not configured it must present a setup modal for Hub URL, API key, and project ID.
Requirement FR-UI-002 The shell must provide a top status bar (project name; live counts for messages/tasks/questions/agents; context warnings; light/dark toggle) and a left sidebar with these pages, grouped WORK / COMMUNICATION / OBSERVE: overview, messages, tasks, questions, activity, logs, agents, jobs, quality, instructions. Sidebar badge counts must come from live queries.

8.2 Data layer

Requirement FR-UI-003 All server state must be fetched through a query-cache layer keyed per resource, using a single authenticated fetch helper that adds the Authorization header, sets JSON content type, warns on 401, and throws typed errors on non-OK. Config/preferences (API key, Hub URL, project ID, theme, light/dark mode) must be held in a small global store, persisting session data to sessionStorage and preferences to localStorage.
Requirement FR-UI-004 The SSE stream must invalidate the relevant query caches per event type: message/read → messages+status; task events → tasks+status; question events → questions+status; heartbeat → tasks+agents; context warning → agents; agent session changed → that agent's sessions. In the reference implementation the central SSE hook owns those invalidations, while session_synced is handled inside the useAgents query (refetch on that event) rather than by the central hook, and agent_output/log_event are consumed to append to the live output and activity/log views (streaming) rather than to invalidate a cache. Live-log and activity views must update from the stream.

8.4 Agent chat & session routing

Requirement FR-UI-005 The agent chat panel must support two session modes — new and resume — via a session selector whose options are a "New conversation" entry plus the agent's existing sessions. Sending a message must POST /api/v1/agent/trigger with {agent, message, session_mode, session_id?}.
Requirement FR-UI-006 Session-routing behavior must follow this algorithm to prevent messages leaking into the wrong session:
  1. If sessions exist and the user has not explicitly chosen "New conversation", auto-select the most recent session and set mode to resume.
  2. When the user clicks "New conversation", set mode to new, select the new-session sentinel, clear local messages, and record the current output-line index as the new-session baseline.
  3. On send, insert an optimistic user bubble immediately, then trigger; on success refetch history, on failure remove the optimistic bubble.
  4. While in new mode, watch only output lines that arrived after the baseline; when a real session ID is detected, switch to resume for that session.
  5. In resume mode, merge server chat history with local messages, de-duplicating optimistic temp messages; do not replace local state with empty history.
Warning Because React re-renders are asynchronous, the send handler must read the current session mode and selected session from refs (not closure-captured state) to avoid a stale-closure bug that routes a new-chat message to the previous session.

12.Regeneration Guide#

This section tells an implementing LLM the recommended order to rebuild the project and how to verify each phase. Build bottom-up: shared model first, then the subsystems that depend on it. For a fully enumerated, trackable task backlog and a repeatable per-task loop, see §14 Development Order & Cycle.

  1. Constants & domain model (§3). Implement the entities, ID generation, validation, and the valid-value sets (statuses, priorities, runners, roles, modes). Verify: unit tests reject invalid status/priority/agent-name and accept valid ones.
  2. Locking & local transport (§5). Implement the file mutex, atomic writes, and the Local transport implementing the six-method interface. Verify: create/list/update a task and send/receive/archive a message round-trip on disk under concurrent access.
  3. CLI (§4). Implement agentweave.yml parsing/validation, the command surface, runners, watchdog, and the .agentweave/ layout + managed .gitignore block. Verify: init → task create → task update → status produces the expected state; doctor reports readiness.
  4. Git & HTTP transports (§5). Add the orphan-branch transport (plumbing only) and the stdlib HTTP transport. Verify: transport status reflects each; git transport never touches the working tree.
  5. Hub backend (§7). Implement the DB schema + migrations, auth, all REST endpoints, SSE manager with the exact event vocabulary, and the scheduler. Verify: the Hub tests pass; POST /messages//tasks//questions return 201 and broadcast SSE; /health and localhost-only /setup/token behave per §7.2.
  6. MCP servers (§6). Implement the CLI-side (transport-backed) and Hub-side (REST-backed) MCP servers with the shared tool set. Verify: get_inbox auto-archives; Hub-only tools error under local transport.
  7. Hub UI (§8). Implement the shell, config bootstrap, query hooks, SSE cache invalidation, components, and the agent-chat session-routing algorithm. Verify: the acceptance flows in §8 hold — especially that a new chat does not route into a previous session.
  8. Logging, roles, docs (§9, §10, §11). Wire the dual logging handlers, multi-role assignment with guide-copy, and the MkDocs site. Verify: events appear in events.jsonl and (with HTTP transport) in the Hub Logs view; roles set writes roles.json and copies guides.
Requirement FR-REGEN-001 A regenerated implementation is conformant when every FR-* requirement in the Requirements Index is satisfied and the per-phase verifications above pass. Behavioral equivalence to the reference implementation should be checked against the CLI and Hub test suites described in §11.3.
Note Maintaining this specification. This document is the source of truth (FR-META-001), so it must not be allowed to drift from the code. Recommended process: (1) any change set that alters a normative value set — the CLI command surface, agentweave.yml keys, runners, MCP tool names, REST endpoints, SSE event names, or DB columns/indexes — updates this document in the same change set; (2) bump the header version badges and add a row to the specification changelog; (3) record any not-yet-reconciled difference as an Open Issue rather than deleting the affected text; (4) keep the Requirements Index traceability column pointing at real, passing tests. 14.Development Order & Cycle#

This section turns the coarse phases of the §12 Regeneration Guide into a granular, trackable backlog and a repeatable per-task working loop, so an AI (or human) can build AgentWeave incrementally — always with a runnable increment and a binary acceptance check at each step. §12 remains the high-level phase order; this section is its enumerated expansion.

14.1The development cycle#

To build the project, repeat the following loop until every task is done:

  1. Select a task. Choose the lowest-numbered task whose dependencies are all done. Tasks marked [P] that share the same satisfied dependencies may be built in parallel.
  2. Ground in the spec. Read the sections named in the task's Satisfies column (its FR-* IDs) and any entities in §3 it touches. Resolve every referenced term before writing code.
  3. Implement in scope. Build only what the task describes. Do not add speculative behavior; unspecified details must not be silently invented — raise them as an Open Issue instead.
  4. Verify. Run the task's Acceptance check. It is a binary pass/fail gate; the task is not done until it passes.
  5. Reconcile. If the implementation reveals that the spec is wrong or incomplete, apply FR-META-001: update the affected spec text (and the changelog) in the same change set, or record an Open Issue.
  6. Record & commit. Mark the task done and commit with a message that references the task ID and the FR-IDs it satisfies (e.g. T-018: DB models + migrations (FR-HUB-006)).
  7. Gate at milestones. When the last task of a milestone completes, run that milestone's gate (§14.4). Do not start the next milestone until the gate passes.
Note The loop is test-anchored: every acceptance check maps to a real test from the §13 traceability column, so "done" always means "verified", never "written".
